1. 测试原理

本测试用于评估多核 CPU cache coherence 子系统（snoop filter / directory / mesh interconnect） 在不同 working-set 大小下的行为，特别是：当被多个 reader 共享的 cache line 总量超过 snoop filter 容量时， 是否触发 back-invalidation（回写失效），从而引入显著的延迟悬崖（"cliff"）。

线程角色与流程

  • Writer（默认 core 0）：每轮迭代对 N 条 cache line 依次写入新版本号， 记录 t_send = rdtsc()，等待所有 reader 在共享 slot 中 ack 完毕， 记录 t_done，得到 fan-out 延迟 = t_done − t_send。
  • Readers（其余所有核）：在 N 条 cache line 上忙等版本号变化， 看到最后一条更新后写入 ack，并记录 t_recv，得到 write-to-observe 延迟 = t_recv − t_send（依赖 invariant TSC 同 socket 同步）。

为什么能观察 snoop filter 容量

当 N × 64 B 远小于 LLC 中 snoop filter 能跟踪的容量时，所有 reader 的 cache line 都能被精确跟踪，writer 的写入仅对持有副本的 reader 发出失效——延迟低且抖动小。

当 N 增大到超过 snoop filter 跟踪容量后，filter 必须驱逐已跟踪的条目； 被驱逐的 line 在 reader 端被强制失效（即使逻辑上仍被使用），writer 的下一次写入 会落到 LLC/内存路径，触发跨 die / 跨 socket snoop 广播——延迟显著上升、p99 抖动放大。 这就是图中观察到的 cliff（性能悬崖）。

不同微架构的 snoop filter 实现差异巨大：Intel Skylake-SP/Ice Lake-SP 使用 mesh + per-CHA snoop filter；AMD Zen 使用 CCX/CCD 内 L3 + 跨 CCD probe filter； ARM Neoverse 使用 SLC + snoop filter。cliff 出现的位置即间接反映了 filter 的跟踪容量。

4. 跨平台对比图

4.1 Writer fan-out 延迟（中位数 p50）

这是核心指标。曲线水平段对应 snoop filter 命中区间；斜率显著上升的位置即 cliff 点。曲线越靠下表示 coherence 路径越快。

4.2 Writer 尾延迟（p99）

p99 反映最差 1% 情况下的延迟。snoop filter 压力大时尾延迟会显著放大，是判断系统是否仍在"线性可扩展区"的重要信号。

4.3 Reader write-to-observe 延迟

展示跨所有 reader 核的中位数曲线，附带 10–90 分位带。区间越宽说明不同 reader 核（如同 die / 跨 die / 跨 socket）之间差异越大。

4.4 尾延迟放大比 p99 / p50

该比值 > 2 通常意味着出现了显著抖动源（snoop 冲突、调度、C-state）。在 cliff 区域附近此比值往往率先抬升，可作为 cliff 的"前兆指标"。

4.5 归一化曲线（消除绝对频率差异）

每条曲线除以自身在最小 working-set 时的 p50 基线，使不同平台在同一坐标系下对比 cliff 形态与位置，避免被 TSC 频率或基础延迟差异干扰判断。

7. 如何得出分析结论

建议的分析步骤：
  1. 看 4.1 与 4.5：归一化曲线最先跨过 2× 的工作集字节数即近端 snoop filter 容量， 跨过 10× 的位置即外层 coherence 域容量。
  2. 看 4.4：在 cliff 之前若 p99/p50 已 > 2，说明系统抖动源不仅是 snoop， 还可能含有 C-state、SMI、频率切换；建议固定频率 + 关闭 turbo 重测。
  3. 看 4.3：若 reader 10–90 分位带很宽，说明拓扑差异显著（跨 die / 跨 socket）； 在 6 节的 per-core 热图中可进一步定位"远端 reader"。
  4. 跨平台对比：水平段更低表示基础 coherence 路径延迟更小（mesh hop / die-to-die fabric 更快）； cliff 更靠右表示 snoop filter 容量更大；放大倍数更小表示退化更平滑、对真实工作负载更友好。
  5. 同 socket 测试结论才直接可比；跨 socket 时请先用 lscpu -e 与 numactl -H 对齐拓扑， 再判断 cliff 是否来自跨 socket 路径。
注意事项：
  • 请确认 invariant_tsc 已启用，且 writer 与 reader 同 socket，否则 reader 延迟数值不可信。
  • SMT（Hyper-Threading）开启时，writer 与其 sibling 的"reader"走 L1/L2，并非真正的跨核 snoop； 建议在测试前关闭 SMT 或排除 sibling 核。
  • 大 working-set（例如 > 32 MB）可能被 LLC 完全踢出，曲线右段同时混合了 DRAM 访问开销， 可与 perf stat -e LLC-load-misses 一起观察。
